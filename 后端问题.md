## 🚀 后端启动步骤（Agent 请执行）

### 1. 确认后端项目位置

后端项目应该是一个 Express.js + MongoDB 的项目，入口文件通常是 `server.js`、`app.js` 或 `index.js`，监听端口 **8080**。

请在包含 `package.json`（含 express/mongoose 依赖）的目录下执行以下操作。

### 2. 启动后端

```bash
# 进入后端项目目录
cd <后端项目路径>

# 安装依赖（如果 node_modules 不存在）
npm install

# 启动服务
npm start
# 或者
node server.js
```

### 3. 验证后端启动成功

```bash
curl http://localhost:8080/health
# 应返回: {"status":"ok"}
```

### 4. 验证 API 前缀可访问

```bash
curl http://localhost:8080/v1/api/versions
# 应返回版本列表 JSON
```

### 5. 前端已就绪

前端 Vite dev server 已启动在 **http://localhost:5173**，会自动代理 `/v1/api` 到 `http://localhost:8080`。

架构：
```
浏览器 → http://localhost:5173 (Vite) → proxy /v1/api → http://localhost:8080 (后端)
```

连接测试方法见 `API-backend.md` 第十二章。

### 4. 后端服务器未启动（P0 必须修复）

**现象**：前端页面所有 API 请求报 `ERR_CONNECTION_REFUSED`

```
POST http://localhost:5173/v1/api/libraries net::ERR_CONNECTION_REFUSED
GET http://localhost:5173/ net::ERR_CONNECTION_REFUSED
```

**原因分析**：
- 前端 `.env` 配置 `VITE_API_BASE_URL=/v1/api`（相对路径）
- Vite dev server（端口 5173）通过 proxy 转发 `/v1/api` → `http://localhost:8080`
- 错误 `ERR_CONNECTION_REFUSED` 说明**端口 8080 上的后端服务没有运行**
- 同时 Vite dev server（端口 5173）也可能未启动

**修复步骤**：
```bash
# 1. 先启动后端服务器（端口 8080）
cd <后端项目目录>
npm start  # 或 node server.js

# 2. 验证后端已启动
curl http://localhost:8080/health
# 应返回: {"status":"ok"}

# 3. 再启动前端开发服务器
cd uni-preset-vue-vite
npm run dev:h5
# Vite dev server 将启动在端口 5173，并自动代理 /v1/api 到 localhost:8080
```

**当前架构**：
```
浏览器 → localhost:5173 (Vite dev server) → proxy /v1/api → localhost:8080 (后端)
                                                   /health → localhost:8080 (后端)
```

> ⚠️ 两个服务都需要启动：Vite dev server（前端）和后端 server（API）。

### 7. 创建卡牌时中文请求体解析失败（P0）

**现象**：连续创建多张含中文的卡牌时，第二次请求报 JSON 解析错误。

```bash
# 第一次创建 — 成功（但存储后乱码，见#6）
curl -X POST .../cards -d '{"jpName":"青眼の白龍","cnName":"青眼白龙",...}'
# → {"id":"...","serial":"TEST01-0001","img":""}

# 第二次创建 — 失败！
curl -X POST .../cards -d '{"jpName":"真紅眼の黒竜","cnName":"真红眼黑龙",...}'
# → {"error":"Unexpected token � in JSON at position 42"}
```

**原因**：`express.json()` 在解析含中文的 JSON body 时，如果 `Content-Type` 未指定 charset，可能使用错误的编码解析，导致 JSON 格式损坏。

**修复**：
```javascript
app.use(express.json({ charset: 'utf-8', limit: '10mb' }))
app.use((req, res, next) => {
  res.setHeader('Content-Type', 'application/json; charset=utf-8')
  next()
})
```

### 8. DELETE 删除库接口返回 404（P0）

**现象**：`DELETE /libraries/:code` 始终返回 404 `{"error":"Not found"}`，即使库确实存在。

```bash
# 库确实存在
curl http://localhost:8080/v1/api/libraries/TEST01?type=local
# {"code":"TEST01","id":"TE00ST01","name":"TEST01","type":"local","cardCount":1}

# 但删除失败
curl -X DELETE "http://localhost:8080/v1/api/libraries/TEST01?type=local"
# {"error":"Not found"}  HTTP 404

curl -X DELETE "http://localhost:8080/v1/api/libraries/TEST01"
# {"error":"Not found"}  HTTP 404
```

**测试了 3 种 URL 格式均失败**：
- `DELETE /libraries/TEST01?type=local` → 404
- `DELETE /libraries/TEST01` → 404
- `DELETE /libraries/TEST01?type=cloud` → 404

**原因**：后端 DELETE handler 中的数据库查询可能使用了错误的字段名或查询条件，导致找不到库记录。

### 9. Transfer 转移后卡牌 ID 变更（P2）

**现象**：上传转移成功后，卡牌的 `id` 发生变化。下载时需要新 ID 才能成功。

```
本地卡牌 ID:  6a360fb2d8708952ffef1eb0
       ↓ upload
云端卡牌 ID:  6a360fbdd8708952ffef1ec4  (新ID!)
       ↓ download (用旧ID)
{"error":"Some cards not found"}    (失败!)
       ↓ download (用新ID)
{"success":true}                     (成功)
```

**影响**：前端在转移后会重新调用 `fetchCards()` 刷新列表，自动获取新 ID，所以功能上不受影响。但 ID 变更不符合"转移"的语义（应该是移动而非复制+删除）。

**建议**：Transfer 操作应保留原卡牌的 `_id`，只更新 `libraryCode` 和 `serial` 字段。

**测试时间**：2026-06-20

```
POST http://localhost:5173/v1/api/vision/recognize 401 (Unauthorized)

2. 或者改为使用用户 JWT Token 认证（和其他接口一致）

> 前端已实现降级：后端 401 后自动切换到豆包 API，功能不受影响。

### 12. Transfer 转移返回 500（P0）

```
POST http://localhost:5173/v1/api/libraries/transfer/upload 500 (Internal Server Error)

2. 或者后端在 transfer handler 中做类型转换

### 13. 本地库/云端库数据隔离 — user_id 设计（P0 新建）

**测试时间**：2026-06-20

**需求**：
- 本地库 = 用户私有的库，只能自己看到和操作
- 云端库 = 所有人共享的库，所有人都能访问
- 两者数据都存储在云端数据库，通过 `user_id` 区分

**数据库设计**：

`libraries` 表需要添加字段：
```
user_id: String   // 本地库 → 当前用户 ID | 云端库 → null
```

`cards` 表需要添加字段：
```
user_id: String   // 本地库中的卡牌 → 当前用户 ID | 云端库中的卡牌 → null
```

**接口行为变更**：

| 接口 | 当前 | 需要改为 |
|------|------|----------|
| `GET /libraries/local` | 返回所有本地库 | 只返回当前 `user_id` 的库 |
| `GET /libraries/cloud` | 返回所有云端库 | 返回 `user_id = null` 的库 |
| `GET /libraries/:code/cards?type=local` | 返回库内卡牌 | 只返回当前用户的卡牌 |
| `POST /libraries` (type=local) | 创建库 | 自动绑定 `user_id = 当前用户` |
| `POST /libraries` (type=cloud) | 创建库 | `user_id = null` |
| `POST /cards` (本地库) | 创建卡牌 | 自动绑定 `user_id = 当前用户` |
| `POST /cards` (云端库) | 创建卡牌 | `user_id = null` |
| `DELETE /libraries/:code?type=local` | 删除库 | 需校验 `user_id` 所有权 |
| `DELETE /cards/:id` | 删除卡牌 | 需校验 `user_id` 所有权 |

**鉴权中间件**：所有本地库操作需要通过 `req.user.id` 过滤 `user_id`。
